Binary Protocol

서버와의 실시간 통신 채널. WebSocket 위에 커스텀 바이너리 프레임을 사용합니다. 서버 ProtocolCommand.java와 동일한 코드 맵을 Android가 ProtocolCommand.kt로 미러링합니다.

1. 프레임 구조

┌─ 2 bytes ─┐┌─ 3 bytes ─┐┌─ 3 bytes ─┐┌─── N bytes ───┐
│  command  ││   length  ││  invokeId ││  JSON body    │
│  (Big-E)  ││   (Big-E) ││   (Big-E) ││  UTF-8        │
└───────────┘└───────────┘└───────────┘└───────────────┘
   0x0201        body length   요청 ID      {"...":"..."}
   (SendChat)                  (Push=0)
  • command — ProtocolCommand.code (2 bytes). 0x0101=Login, 0x0A01=HI, 0x0280=SendChatEvent ...
  • length — body 길이 (3 bytes)
  • invokeId — 요청-응답 매칭용 (3 bytes). Push는 0
  • body — UTF-8 JSON 문자열

인코딩/디코딩 파일

  • BinaryFrameCodec.kt — encode/decode
  • BinarySocketClient.kt — OkHttp WebSocket + coroutine loops
  • SocketSessionManager.kt — 세션 관리 (581줄)
  • ProtocolCommand.kt — enum (100+ 명령)

2. 요청/응답 매칭

sequenceDiagram participant R as Repository participant M as SocketSessionManager participant S as Server R->>M: sendFrame(cmd, body) M->>M: id = ++invokeId M->>M: pending[id] = CompletableDeferred() M->>S: [cmd][len][id][body] S-->>M: [cmd][len][id][body] M->>M: pending[id].complete(body) M-->>R: await() → body Note over S,M: Push (invokeId=0) S-->>M: [cmd][len][0][body] M->>M: isPushEvent(cmd) → PushEventHandler

4. Push 이벤트 처리 파이프라인

PushEventHandler가 invokeId==0 프레임을 받아 명령 코드별로 분기:

// 수신 후 분기
when (command) {
    ProtocolCommand.SEND_CHAT_EVENT.code -> handleSendChatEvent(data)
    ProtocolCommand.READ_CHAT_EVENT.code -> handleReadChatEvent(data)
    ProtocolCommand.ICON_EVENT.code      -> handleIconEvent(data)      // presence
    ProtocolCommand.NOTIFY_EVENT.code    -> handleNotifyEvent(data)
    ProtocolCommand.RETRIEVE_MESSAGE_EVENT.code -> handleRetrieveMessageEvent(data)
    // ... 기타
}

각 핸들러는 로컬 DB에 반영 후 forwardPushToReact(command, data)로 Web에 {event:"neoPush", command, data} 전달.
