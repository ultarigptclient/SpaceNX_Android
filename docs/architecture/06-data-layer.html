06. Data Layer & Repository

Room + SQLCipher 기반 로컬 DB, Retrofit REST, Binary Socket 세 소스를 Repository가 종합합니다. 이 페이지는 레이어 전체 그림과 Repository 역할 분담을 설명합니다. DB 스키마 상세는 Data 섹션을.

1. 레이어 전체 그림

flowchart TD UI[UI / Bridge Handler] VM[ViewModel] R[Repository] DAO["Room DAO"] DB[("SQLCipher-encrypted Room DB")] API["Retrofit API"] S[SocketSessionManager] UI --> R VM --> R R --> DAO DAO --> DB R --> API R --> S

3. Room + SQLCipher

3-1. 6개 DB 분리

projectproject_{userId}.db사용자별ProjectEntity, IssueEntity, ChatThreadEntity, ThreadCommentEntity 등

3-2. 암호화

  • 모든 DB는 SQLCipher로 암호화
  • Passphrase: AmCodec.decryptSeed()로 앱/디바이스 고유 값에서 유도
  • HybridWebMessengerApp.onCreate()에서 미리 warmUpPassphrase() 호출 (첫 DB 접근 지연 방지)

3-3. 마이그레이션 정책

현재 모든 DB가 fallbackToDestructiveMigration() 사용 중. 스키마 변경 시 기존 데이터가 전부 날아갑니다. 개발 단계에서는 OK지만, 프로덕션 배포 시점에는 Migration 구현 필요.

4. Repository 패턴 예시

4-1. sync 흐름 (REST → Room)

suspend fun syncChat(userId: String): JSONObject {
    val lastSyncTime = commonDao.getSyncMeta("chat_last_sync_time")?.value?.toLongOrNull() ?: 0L
    val body = JSONObject().apply {
        put("userId", userId)
        put("lastSyncTime", lastSyncTime)
    }

    val response = ApiClient.postJson(
        appConfig.getEndpointByPath("/comm/syncchat"),
        body,
        appConfig.getSavedToken()
    )

    // DB 반영
    val chats = response.optJSONArray("chats") ?: return response
    val entities = (0 until chats.length()).map {
        ChatEntity.fromJson(chats.getJSONObject(it))
    }
    withContext(Dispatchers.IO) {
        chatDao.insertAll(entities)
    }

    // lastSyncTime 갱신
    val newTime = response.optLong("lastSyncTime", System.currentTimeMillis())
    commonDao.upsertSyncMeta(SyncMetaEntity("chat_last_sync_time", newTime.toString()))

    return response
}

4-2. Push 이벤트 → Room 적용

// PushEventHandler.handleSendChatEvent
private suspend fun handleSendChatEvent(data: JSONObject) {
    val chat = ChatEntity.fromJson(data)
    dbProvider.getChatDatabase().chatDao().upsert(chat)

    // unreadCount 재계산
    channelRepo.recalcChannelUnread(chat.channelCode)

    // React에 전달
    forwardPushToReact("SendChatEvent", data)
}

6. 오프셋/이벤트 기반 sync 개념

채팅/쪽지/알림은 "마지막 동기화 이후의 변경분만 가져오기" 전략을 씁니다.

flowchart LR A["앱 시작 / sync 진입"] --> B["commonDao에서
lastSyncTime or lastEventId 조회"] B --> C["서버에 POST
/sync{도메인} {userId, lastXxx}"] C --> D["응답: {items:[...], newLastXxx}"] D --> E["Room upsertAll(items)"] E --> F["commonDao.upsertSyncMeta(newLastXxx)"] F --> G["notifyReact('{domain}Ready')"]

각 Entity의 offset 갱신은 updateCrudOffset() 공용 헬퍼에서 처리합니다 (BridgeContext에 노출).

7. REST + Socket 이원 운영 패턴

같은 명령에 대해 REST와 Socket을 둘 다 쓰는 경우 (예: changeStatus). 순서는 설정으로 결정:

"changeStatus" -> scope.launch {
    val success = if (PRESENCE_REST_PRIORITY) {
        tryRest() || trySocket()
    } else {
        trySocket() || tryRest()
    }
    // 성공 시 본인 상태 즉시 반영 (서버는 본인에게 ICON_EVENT 안 보냄)
}

8. 캐시 전략

  • UserNameCache — userId → userName 메모리 맵. org.db에 없을 때 buddy.db에서 fallback
  • activeChannelCode — 현재 열린 채널을 BridgeContext에 보관 (in-app 알림 억제용)
  • configCache — AppConfig에 서버 설정 전체를 메모리 맵으로 유지 (REST 엔드포인트 등)

9. 로그아웃 시 데이터 처리

현재는 SQLCipher DB 파일은 유지하되, 메모리 캐시만 초기화합니다. 다른 사용자로 로그인하면 chat_{userId}.db 파일이 새로 열립니다.

// AuthRepository.requestLogout()
sessionManager.disconnect()
appConfig.clearTokens()         // access/refresh 토큰만 삭제
userNameCache.clear()
channelRepo.clearMemoryCache()
