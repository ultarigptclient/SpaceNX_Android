Hybrid-Talk v2 아키텍처 문서

웹앳톡(Web@Talk) Android 하이브리드 메신저의 아키텍처, 데이터 레이어, 통신 프로토콜, 주요 플로우를 한 곳에 모은 개발 문서입니다. 이 앱은 네이티브 Kotlin 코어가 인증·DB·소켓을 담당하고, WebView 내부 React SPA가 UI를 그리는 구조입니다. 둘 사이를 JavaScript Bridge로 연결합니다.

처음 보신다면 이 순서로 읽으세요
  1. 전체 흐름도 & 구조 — 앱 시작부터 WebView 로드까지 전체 그림
  2. 통신 구조 (3축) — Bridge / Binary Socket / REST 어떻게 쓰는지
  3. 세션 & 로그인 흐름 — 수동·자동 로그인, HI 핸드셰이크
  4. DB 개요 & ER — 6개 DB 구조, 엔티티 관계
  5. Bridge & Handlers — 실제 코드 작성 시 가장 자주 건드릴 부분

1분 요약

  • 앱 타입: WebView + 네이티브 하이브리드 (React SPA를 원격 URL로 로드)
  • 빌드: Gradle 8.11.1, Kotlin 2.1.10, Java 17, compileSdk 35 / minSdk 29
  • DI: Hilt (@HiltAndroidApp, @AndroidEntryPoint, @HiltViewModel)
  • DB: Room + SQLCipher 6종 (common / org / chat / message / noti / setting / project)
  • 통신
    • REST Retrofit + OkHttp (:18019)
    • Binary 커스텀 WebSocket 프레임 프로토콜 (:18000)
    • Bridge WebAppBridge.postMessage() ↔ window._${action}Resolve()
    • Push FCM + Binary Push 프레임 → window.postMessage({event:"neoPush"})
  • 아키텍처 의도: Clean Architecture 기반으로 패키지 설계 → 실제 구현은 MVVM에 가깝게 수렴
  • 테스트: 의존성은 선언되어 있으나 실제 테스트 파일 0개 (개선 대상)

리팩토링 대상 & 개선 포인트

현재 코드에는 몇 가지 알려진 거대 파일이 있습니다. 새로 합류한 멤버는 이 지점을 먼저 이해하면 좋습니다.

파일라인 수이슈

MessageRepository.kt552sync + CRUD + prefetch + mutex

계획된 분리 방향 (참고)
  • MainActivity → WebViewManager / PushHandlerRegistry / NetworkMonitor로 쪼개기
  • ChannelRepository → ChannelSyncRepository / ChatRepository / ChannelMemberRepository
  • Handler를 @Inject 기반으로 전환 (현재는 BridgeDispatcher에서 직접 생성)
  • AppConfig(354줄) → CredentialManager / EndpointResolver / ConfigCache로 책임 분리
  • 통합 에러 리포팅 + 테스트 하네스 추가
