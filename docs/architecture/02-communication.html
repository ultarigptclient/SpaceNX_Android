02. 통신 구조 (3축)

이 앱은 3개의 독립된 통신 채널을 병행합니다. 각각 역할이 다릅니다: Bridge React↔Native, REST 대부분의 요청/응답, Binary Socket 실시간 Push와 일부 빠른 명령. 여기서 3축의 역할·구조·구현 파일을 정리합니다.

1. 왜 3축인가?

채널용도프로토콜연결성
Bridge React SPA와 네이티브 사이의 함수 호출 postMessage + evalJavascript 항상 연결 (WebView 내부)
REST 대부분의 CRUD, sync, 로그인, 업로드 HTTPS + JWT Stateless (매 요청마다 토큰)
Binary 실시간 Push, HI 핸드셰이크, Presence, 일부 빠른 명령 커스텀 바이너리 프레임 over WebSocket 지속 연결 (session-scoped)
한 줄 요약: "가능하면 REST, 실시간이어야 하면 Socket, UI 연결은 Bridge."

2. Bridge JS ↔ Native

2-1. 엔트리포인트: WebAppBridge

WebView에 window.AndroidBridge라는 이름으로 노출됩니다. React가 AndroidBridge.postMessage(JSON.stringify({action, ...params}))를 호출하면 이 클래스의 @JavascriptInterface fun postMessage(message: String)가 호출됩니다.

class WebAppBridge(private val dispatcher: BridgeDispatcher) {
    companion object {
        const val NAME = "AndroidBridge"
    }

    @JavascriptInterface
    fun postMessage(message: String) {
        val json = JSONObject(message)
        val action = json.optString("action", "")
        val params = mutableMapOf<String, Any?>()
        val keys = json.keys()
        while (keys.hasNext()) {
            val key = keys.next()
            if (key != "action") params[key] = json.opt(key)
        }
        dispatcher.dispatch(action, params)
    }
}

2-2. 라우팅: BridgeDispatcher

거대한 when(action) 블록으로 8개 Handler에 위임합니다. 하나의 action이 여러 개 그룹으로 묶여 같은 Handler로 가기도 합니다. 자세한 건 Bridge & Handlers에서.

2-3. 응답: evalJavascript

모든 응답은 JS 전역 함수 호출로 돌려줍니다. action 이름이 login이면:

// 성공
override fun resolveToJs(action: String, data: JSONObject) {
    evalJs("window._${action}Resolve && window._${action}Resolve('${esc(data.toString())}')")
}
// 실패
override fun rejectToJs(action: String, errorMessage: String?) {
    val err = JSONObject().put("errorCode", -1).put("errorMessage", errorMessage ?: "")
    evalJs("window._${action}Reject && window._${action}Reject('${esc(err.toString())}')")
}

React 측은 Promise로 감싸서 사용합니다:

function nativeSend(action, params) {
    return new Promise((resolve, reject) => {
        window[`_${action}Resolve`] = (data) => resolve(JSON.parse(data));
        window[`_${action}Reject`] = (err) => reject(JSON.parse(err));
        window.AndroidBridge.postMessage(JSON.stringify({action, ...params}));
    });
}

2-4. 서버 Push를 Web으로 전달

Binary 소켓으로 들어온 Push 이벤트는 DB에 반영한 뒤 Web에 window.postMessage({event:"neoPush", command, data})로 재전송합니다. Push 이벤트 플로우 참고.

3. REST Retrofit

3-1. 구성

  • 인터페이스 7종: AuthApi, OrgApi, BuddyApi, ChannelApi, CommApi, StatusApi, UpdateApi
  • 공통 클라이언트: ApiClient — Retrofit 인스턴스 관리 + postJson() 범용 호출 + 파일 업로드
  • Base URL: https://neo.ultari.co.kr:18019/api/ (AppConfig.getRestBaseUrl() — syncConfig 응답 기반 가변)
  • 인증: JWT Bearer (Authorization: Bearer {accessToken})
  • 갱신: 401 수신 시 refreshToken으로 자동 재발급 후 재시도

3-2. 호출 패턴

// 특정 인터페이스 메서드
val response = authApi.login(LoginRequest(loginId, password))

// 범용 JSON POST (NeoSendHandler에서 주로 사용)
val result = withContext(Dispatchers.IO) {
    ApiClient.postJson(
        appConfig.getEndpointByPath("/comm/sendchat"),
        body,
        appConfig.getSavedToken()
    )
}

3-3. Endpoint resolution

syncConfig 응답으로 받은 ultari.rest.endpoints.* 값이 AppConfig.configCache에 저장되고, Repository가 이 맵을 거쳐 실제 URL을 얻습니다. DB에 저장된 값이 없으면 strings.xml의 기본값으로 fallback.

주의: SSL 검증 비활성화
현재 개발 편의를 위해 Trust Manager가 스텁 상태입니다. 배포 전 정책 결정이 필요합니다 (CLAUDE.md에도 명시).

4. Binary Socket 0x0XXX 프레임

4-1. 프레임 포맷

┌─ 2 bytes ─┐┌─ 3 bytes ─┐┌─ 3 bytes ─┐┌─── N bytes ───┐
│  command  ││   length  ││  invokeId ││  JSON body    │
│  (BE)     ││   (BE)    ││   (BE)    ││  UTF-8        │
└───────────┘└───────────┘└───────────┘└───────────────┘
  • command: ProtocolCommand.code 값 (0x0101=Login, 0x0201=SendChat, 0x0A01=HI ...)
  • length: body 길이 (bytes)
  • invokeId: 요청ID (응답 매칭). Push 이벤트는 0
  • body: JSON UTF-8

4-2. 구현 파일

파일역할
BinarySocketClient.ktOkHttp WebSocket 연결 + 3 coroutines (read / send / keepAlive)
BinaryFrameCodec.kt프레임 인코딩/디코딩 (encode(), decode())
ProtocolCommand.ktenum — 100+ 명령코드 정의
SocketSessionManager.kt세션 생명주기: 연결·HI·재연결·프레임 라우팅 (581줄)

4-3. 요청/응답 매칭 (invokeId)

sequenceDiagram participant R as Repository participant M as SocketSessionManager participant S as Server R->>M: sendFrame(cmd, body) M->>M: invokeId++ M->>M: pending[invokeId] = CompletableDeferred() M->>S: [cmd][len][invokeId][body] S-->>M: [cmd][len][invokeId][body] M->>M: pending[invokeId].complete(body) M-->>R: deferred.await() → body Note over S,M: Push (invokeId=0) S-->>M: [cmd][len][0][body] M->>M: isPushEvent(cmd) → PushEventHandler.handle()

4-4. Keep-Alive

30초마다 NOOP(0x0A09) 프레임 전송. 연결이 끊기면 백오프 재연결.
